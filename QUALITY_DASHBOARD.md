## Open quality issues — by category

### `wallet-api` (15 open)

**Authorization / authz**
- [#2](https://github.com/jeffgicharu/wallet-api/issues/2) — AdminController endpoints accessible to any authenticated user (HIGH)
- [#20](https://github.com/jeffgicharu/wallet-api/issues/20) — `GET /api/wallet/transactions/{ref}` returns any user's transaction (HIGH)

**Auth / cryptography**
- [#4](https://github.com/jeffgicharu/wallet-api/issues/4) — Default Spring profile contains a hard-coded JWT secret
- [#19](https://github.com/jeffgicharu/wallet-api/issues/19) — JwtTokenProvider uses platform default charset for HMAC key bytes
- [#21](https://github.com/jeffgicharu/wallet-api/issues/21) — No rate limit on `/api/auth/login`

**Persistence / schema**
- [#3](https://github.com/jeffgicharu/wallet-api/issues/3) — Production profile uses Hibernate `ddl-auto: update`
- [#5](https://github.com/jeffgicharu/wallet-api/issues/5) — `spring.jpa.open-in-view=true` triggers warning
- [#11](https://github.com/jeffgicharu/wallet-api/issues/11) — System-wide reconciliation reports `balanced=false` after any activity

**Behavioural / contract**
- [#9](https://github.com/jeffgicharu/wallet-api/issues/9) — PIN lockout counter is rolled back when validation fails
- [#10](https://github.com/jeffgicharu/wallet-api/issues/10) — Idempotent retry on duplicate transfer returns 409 instead of the original transaction
- [#12](https://github.com/jeffgicharu/wallet-api/issues/12) — Transaction reversals are not recorded in the audit log

**Performance**
- [#15](https://github.com/jeffgicharu/wallet-api/issues/15) — Throughput plateaus at ~55 req/s under load (bcrypt suspected)
- [#16](https://github.com/jeffgicharu/wallet-api/issues/16) — Read endpoints degrade to p95 1172 ms at 50 VU (HikariCP pool)
- [#17](https://github.com/jeffgicharu/wallet-api/issues/17) — Workflow test surfaces 0.26 % optimistic-lock 409s on transfer

**Dependencies**
- [#22](https://github.com/jeffgicharu/wallet-api/issues/22) — 22 HIGH+ CVEs (4 CRITICAL) in Spring Boot 3.2.5 transitive deps

This dashboard is a single snapshot. Trend tracking is a planned follow-up:

- Weekly snapshot committed to `quality-trends/YYYY-WW.json` by a scheduled GitHub Action.
- Coverage, mutation, and performance numbers plotted alongside open-issue counts to visualise debt direction.

None of the above lands in this PR. The infrastructure to make it cheap to start is `scripts/quality-snapshot.sh` — a scheduled cron that runs the script and commits its diff is the minimum-viable trend layer.

## How this dashboard updates

```bash
# After running mvn verify + mutation + perf + browsing the wallet-app reports:
bash scripts/quality-snapshot.sh
```

The script reads the various tool outputs and rewrites the metric tables in this file between the `<!-- key:start -->` / `<!-- key:end -->` fences. Idempotent — re-runs replace rather than append. See the script header for data-source paths.

If `wallet-app` is checked out at a non-sibling location, override with `WALLET_APP_DIR=/path/to/wallet-app bash scripts/quality-snapshot.sh`.
